# Use uv and real NautilusTrader for Python validation

The Python evaluator will be managed as a root-level `uv` project targeting Python 3.12 by default, with package metadata allowing only the NautilusTrader-supported Python range. Any final strategy validation path must execute inside NautilusTrader rather than a Nautilus-compatible replay helper. The first real integration path must run a fixture ES RTH 5-minute Versioned Dataset through NautilusTrader's backtest engine with a single-instrument, single-venue minimal Strategy Spec and write a Run Registry record marked as Nautilus Validation. Strategy Specs will execute through a fixed Nautilus Strategy adapter/interpreter rather than generated Python strategy code. Fees and slippage for final validation must be mapped into the Nautilus backtest execution configuration; unsupported cost models should fail fast instead of being applied as post-processing. Formal Run Registry records that feed Fitness Scores, search ranking, or search winners must include real NautilusTrader provenance, including engine/version, environment, dataset, Strategy Spec, instrument/venue/bar type, execution-cost configuration, and produced Nautilus artifacts. NautilusTrader source will be kept as a read-only reference submodule at `third_party/nautilus_trader`, fixed to the same release as the `uv.lock` dependency, starting with `v1.229.0`; runtime imports must resolve from the `uv`-locked package dependency, not the submodule path. Default validation must include unit tests plus a real Nautilus integration smoke test, while fuller search and walk-forward integration can remain an explicit slower command until stable. Lightweight replay code may remain for smoke tests and dataset plumbing, but it must not write formal Run Registry records, produce final Fitness Scores, select search winners, or claim Nautilus Validation.
